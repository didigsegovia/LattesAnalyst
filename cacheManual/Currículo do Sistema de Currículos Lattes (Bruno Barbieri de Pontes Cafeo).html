Análise de Emoções em Plataformas de Codificação Social Modernas sob a Perspectiva das Reações dos Desenvolvedores
Descrição: Neste projeto de pesquisa é proposto o desenvolvimento de estudos empíricos sobre como devesenvolvedores reagem à artefatos produzidos no desenvolvimento de software. Especialmente, pretende-se em analisar como desenvolvedores usam a funcionalidade reactions de plataformas de codificação modernas. Reactions é uma funcionalidade social que teve origem em redes sociais, e.g., Facebook, e expandiu a forma como as pessoas interagiam nestes ambientes. Essa funcionalidade rapidamente ganhou notoriedade e foi adotada por diversas outras plataformas sociais, incluindo plataformas de codificação, como por exemplo, GitHub e Bitbucket. Embora diversos estudos explorem reactions em redes sociais, pouco se sabe sobre o seu impacto no contexto de plataformas de codificação social modernas. Um dos objetivos deste projeto de pesquisa é determinar como desenvolvedores veem essa funcionalidade e a influência desses dados no processo de desenvolvimento de software atual. Para lançar uma luz sobre esse problema propõe-se o desenvolvimento de estudos empíricos a fim de entender melhor como desenvolvedores usam reactions e também de contribuir para a melhora do processo de desenvolvimento nestas plataformas..
Situação: Em andamento; Natureza: Pesquisa.

Investigação sobre Developer Relations (DevRel) em Ecossistemas de Software
Descrição: Organizações que mantêm ecossistemas de software (ECOS), como Apple e Google, precisam atrair e engajar uma massa crítica de desenvolvedores externos para sustentar o ecossistema e atender às demandas dos usuários. Por isso, as organizações têm investido em estratégias de governança como um time de Developer Relations (DevRel). Um time de DevRel consiste em profissionais da organização que planejam, executam e analisam os resultados de ações com as comunidades de desenvolvedores e empresas parceiras dentro do ecossistema. Os profissionais de DevRel atuam com o evangelismo (larga-escala, ou seja, a divulgação dos recursos do ecossistema) e em advocacia (em profundidade, ou seja, proximidade e ligação direta com os desenvolvedores). Isto para estabelecer a sinergia entre expectativas de desenvolvedores externos e objetivos da organização. Porém, isto não é trivial, se a estratégia for insustentável, pode-se incorrer na ?morte? do ecossistema. Esta proposta de projeto considera a investigação de abordagens em torno da governança de desenvolvedores em ECOS a partir da perspectiva de DevRel. As abordagens (p.ex: técnicas, métodos, processos, algoritmos, modelos) de DevRel para a governança de desenvolvedores em ECOS devem ser avaliadas se são adequadas, entendíveis, controláveis e generalizáveis para apoiar estratégias de profissionais de DevRel. Isto por meio de métodos de Engenharia de Software Experimental (estudos quantitativos e qualitativos), de Mineração de Repositórios de Software e de Aprendizagem de Máquina..
Situação: Em andamento; Natureza: Pesquisa.

COMVIP-SC: Combatendo Violações de Padrões de Uso de APIs em Sistemas Configuráveis
Descrição: APIs (Application Programming Interface) têm sido vastamente adotada no desenvolvimento de software. Mineração de Bibliotecas de Software para Melhoria de Qualidade de Código
Descrição: Sistemas de software estão em constante evolução: novas funcionalidades são adicionadas, defeitos são corrigidos e códigos são refatorados. Um sistema que não é atualizado tem o risco de ficar obsoleto e sem relevância. Atualmente, sistemas são comumente parte de ecossistemas de software, que existem geralmente em grandes empresas e comunidades open-source, tais como Eclipse, Android e distribuições Linux. Como qualquer outro software, os que compõem ecossistemas também evoluem com o passar do tempo. Nesses ambientes, a evolução de uma biblioteca de software (comumente utilizadas para reuso de código e aumento de produtividade) pode impactar suas aplicações clientes. A alteração em uma API fornecida por tais bibliotecas pode gerar reações no ecossistema. Na prática, o tamanho do impacto causado pela evolução de APIs é grande e seus efeitos desconhecidos pelos desenvolvedores e clientes, gerando perda de qualidade nos sistemas envolvidos. Visando amenizar esses problemas, neste projeto de pesquisa, planeja-se realizar estudos empíricos para auxiliar tanto os desenvolvedores de bibliotecas quanto os clientes em suas atividades de manutenção e evolução de código..
Situação: Concluído; Natureza: Pesquisa.

Estudos Empíricos e Apoio à Manutenção de Linhas de Produto de Software na Presença de Dependência entre Features
Descrição: eatures são abstrações-chave para o desenvolvimento e manutenção de linhas de produto de software (LPS). Um desafio encarado na manutenção de LPS é o entendimento das dependências que existem entre features. No código-fonte, uma dependência entre features ocorre sempre que um elemento de programa dentro dos limites de implementação de uma feature depende de elementos externos a esta feature. Exemplos são atributos ou métodos definidos na implementação de uma feature, mas utilizados no código responsável pela implementação de outra feature. A medida que desenvolvedores modificam o código-fonte associado com uma feature, eles devem garantir que outras features são consistentemente atualizadas com as novas mudanças. No entanto, a apropriada manutenção de features não é uma tarefa trivial, pois features geralmente não são modularizadas no código-fonte. Como consequência, a implementação de features e suas dependências ficam espalhadas no código-fonte. Dessa forma, dado uma mudança em uma determinada feature, é desafiador revelar quais (partes de) outras features também devem ser alteradas. A manutenção se torna, portanto, um aspecto central e não-trivial do ciclo de vida de LPS. Infelizmente, não existe entendimento no estado-da-arte sobre propriedades estruturais de dependências de features que afetam a propagação de mudanças. Nesse contexto, esse projeto de pesquisa visa a contribuir, principalmente (i) no estudo empírico para entendimento do impacto das dependências entre features em atributos de qualidade referentes à manutenção de LPS, (ii) propor um modelo de representação de informações de dependência entre features de modo a auxiliar o entendimento da organização de tais dependências..
Situação: Concluído; Natureza: Pesquisa.

CARECO - Sistemas de Recomendação para Manutenção Colaborativa de Software
Descrição: O objetivo deste projeto entitulado CAReCo: Cooperação Acadêmica em Sistemas de Recomendação para Manutenção Colaborativa de Software é investigar como sistemas de recomendação podem apoiar efetivamente as tarefas de manutenção de software e contornar as limitações mencionadas anteriormente. Este objetivo será alcançado através de cooperações entre especialistas com atuação destacada em cada uma das áreas relevantes e que fazem parte das instituições integrantes PUC-Rio, UFAL, UFCG e UFAM. A equipe do projeto é formada por pesquisadores de Programas de Pós-graduação nestas quatro instituições, localizadas em três diferentes regiões do país: Sudeste, Nordeste e Norte. Além disso, cada um dos Programas possuem diferentes níveis de maturidade, tanto com relação aos seus cursos de Mestrado, como os de Doutorado. O coordenador geral é o Prof. Carlos José Pereira de Lucena (bolsista produtividade nível 1A do CNPq), que possui ampla experiência na coordenação de projetos multi-institucionais. Manutenção de Linhas de Produtos de Software: Interfaces Explícitas de Características
Descrição: Engenharia de linha de produto de software tem como um dos seus objetivos criar produtos adaptados a partir de um conjunto reutilizável de ativos para satisfazer as necessidades dos clientes. Diversas abordagens de linhas de produto focam na decomposição de software em unidades modulares chamadas features. Features são unidades de comportamento semanticamente coesas de um sistema de software. Duas tarefas recorrentes na manutenção de linhas de produto são a identificação e entendimento de dependência entre features no código-fonte. A identificação apropriada e o entendimento de elementos de programas envolvidos na dependência entre features é essencial para a manutenção independente de uma feature. Tais elementos de programas que contribuem para a comunicação entre features são considerados partes da chamada interface de uma feature. A representação explícita, bem como o entendimento da interface de uma feature possibilita ao desenvolvedor raciocinar de forma adequada sobre a manutenção de features como unidades modulares. O problema é que, no código-fonte, os limites das features não são os mesmos dos limites dos módulos de um programa (ex., pacotes e classes). Nesse caso, o código das features podem se encontrar espalhados ao longo do código-base e, portanto, as interfaces das features não estão explícitas no código-fonte. Além disso, após identificar tais elementos no código, desenvolvedores geralmente devem raciocinar sobre elementos de interfaces de features e abstrair elementos que não úteis para a realização da manutenção. Dessa forma, o principal objetivo deste projeto é a melhoria na representação da interface de features de forma a apoiar a manutenção do código de linhas de produtos..
